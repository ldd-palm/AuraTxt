整体架构方案对比

三种 Solution 结构方案，核心差别在于 GUI 与 CLI 之间的耦合方式

★

方案 A — 分层共享库（推荐）

一个 Solution，三个 Project：

  • AuraTxt.Core（Class Library）— 共享数据模型、config.json 序列化/反序列化、AI 客户端抽象、Google Translate / 有道移植
  • AuraTxt（WPF App）— 系统托盘、全局钩子、弹窗 UI、调用 Core
  • AuraTxt.Cli（Console App）— config-cli.exe，引用 Core，读写同一份 config.json

config.json 存放在 %AppData%\AuraTxt\config.json（Windows 标准位置）

优势

  • JSON 解析逻辑只写一次，GUI 和 CLI 绝对同步
  • Claude Code 在同一 repo 操作 CLI，无需跨 repo
  • 结构清晰，依赖方向单向（WPF/CLI → Core）

劣势

  • Core 库发布时需跟 exe 一起打包（可 PublishSingleFile 合并）
B

方案 B — 运行时 IPC 通信

GUI 和 CLI 通过本地 Named Pipe 或 REST 通信，CLI 发送修改指令，GUI 实时热重载配置，无需重启。

优势

  • 配置修改后 GUI 立即生效，无需重启

劣势

  • 复杂度大幅上升，调试困难
  • 需要 GUI 常驻监听端口，不适合轻量工具
  • 需求文档没有要求热重载
C

方案 C — 单 exe 内置 CLI 模式

只有一个 exe，通过命令行参数切换模式：AuraTxt.exe --cli model --set ... 时进入 CLI 模式，无参数时启动 GUI。

优势

  • 只需分发一个文件

劣势

  • WPF exe 做 CLI 时会闪烁控制台窗口，体验差
  • 两个截然不同的用途混在一起，代码结构混乱
  • 违背需求文档的解耦设计初衷

文字选择检测方案（技术内定）

这个不需要你选择，已确定最优方案：

  • 全局低级鼠标钩子（WH_MOUSE_LL）监听 LButtonUp
  • 触发后 100ms 延迟（等待选区稳定），保存剪贴板旧内容
  • 模拟 Ctrl+C 读取选中文字（通用，覆盖所有应用）
  • 读取完毕后立即恢复旧剪贴板内容（用户无感知）
  • 若选中内容非空，在光标位置弹出动作菜单
